# 项目复盘

## 成果

本次交付覆盖了两条核心路径下共 7 个 Android BDD 场景:

| 路径 | 场景 | 结果 |
|---------|-----------|--------|
| Add Transaction | Expense、Income、Transfer、空金额、自定义类别 | 5 个通过 |
| Transactions | 类型筛选、按历史日期分组 | 2 个通过 |

每个业务场景都完成了首次运行 setup,成功的 Add Transaction 场景同时
校验了 Transactions 中的对应行以及 Home 上的月度汇总。最近一次本地
Allure 运行结果为 7 通过、0 失败、0 跳过。

## 做得好的地方

### 清晰的层级归属

把 Gherkin、Steps、Flows、Pages 与 locator YAML 分离,让失败被定位在
单一层。例如,一次键盘交互变更只影响一个 Page 方法,而不是所有场景。
事务汇总计算也得以作为 Flow 行为而非 UI 选择器逻辑来测试。

### 确定性的 Setup

在每个场景前清空包数据有性能成本,但让 onboarding、自定义类别创建、
汇总基线都可重复。显式输入用户名、货币、预算和短信偏好,也是测试真正
的首次运行路径,而不是用 Skip 绕过。

### 越过保存确认的断言

成功的场景不会在 Add Transaction 表单关闭时就停下。它们会在
Transactions 中校验日期、金额、类别、时间,然后再在 Home 上检查收入、
支出、结余以及预算百分比。这能捕获到只靠 toast 或 Recent Transactions
断言发现不了的集成问题。

### 失败证据

Allure 使用原始的 BDD Feature 与 Scenario 名称。call 阶段失败时,本地
PNG 会被保存,并附加到结果里,不包裹测试体,也不隐藏原始 traceback。

## 比预期更困难的地方

### Flutter 的可访问性语义

部分控件暴露了较强的 accessibility ID,另一些则被合并进大型语义节点,
或者只暴露位置性的文本字段。套件因此包含受限的 XPath 兜底。它们被
隔离在 YAML 中,但对应用布局变更比稳定的语义 ID 更敏感。

### 键盘状态

Android 键盘会同时挡住 onboarding 的预算滑块和 Add Transaction 的控件。
在金额、备注、标签和类别名输入后发送 IME Done 动作,比点击任意坐标
更可靠。

### 横向类别

`New` 类别入口位于横向列表的最右端。创建 `baby cost` 需要受控的滑
动、类别创建、Back 导航,以及选中一个可能因为固定宽度只显示 `baby`
的 chip。

### 过渡竞态

当一个场景内连续添加两笔交易时,第二次的 Home 快捷入口可能命中关闭
中表单上同名的 tab。在保存成功后等待 Home 可见,消除了这一竞态,让
顺序事务也变得可靠。

## 当前的局限

- 每个设备 worker 都是串行的;矩阵模式增加的是设备级并发,并不能在同
  一目标上并行多个 Appium 会话。
- 全量包重置使每个场景比有针对性的 state fixture 更慢。
- iOS locator 与全部 7 个场景已在运行 iOS 26.5 的 iPhone 17 模拟器上
  校验;其他设备 profile 和语言区设置尚未验证。
- 照片附件与更广的 Settings/Analytics 行为不在 7 个场景范围内。
- GitHub 上的 E2E 需要通过 `TRACKIFY_APK_URL` secret 提供可外部下载
  的 APK;没有它时,CI 只跑单元测试和 collection。
- 任务 14 是有意单向的:Excel 更新托管的 Scenario 块,但不把结果回写,
  locator 也不会自愈。
- 任务 13 是顾问式而非自主的。MiniMax 兼容回退已经过实测,但严格的
  JSON 失败仍会降级为 `Unknown`;在调整阈值或策略前需要更多真实失败
  样本。
- `summary.xlsx` 的生成在任务 11 中被刻意延后。

## AI 协助

AI 帮助审阅仓库、把规范翻译为 BDD 与 Page Object、推理 Flutter/Appium
的 locator 行为,并缩短调试周期。每一次建议的交互都会用真实模拟器
对照检查。最有用的反馈来自真实的失败:类别导航、键盘状态、日期/时
间持久化、保存后的竞态 —— 这些都需要从运行中的应用取得证据,而不是
照单全收生成的代码。

任务 13 内部也遵循同样的边界:已知失败形态优先使用确定性特征,只有
证据模糊时才使用 MiniMax。实测已经验证了鉴权、TLS 信任、请求/响应兼
容性以及安全降级,但模型建议仍然是附加在原始失败上的假设,不是自动
判定。

## 后续改进

1. 用更大规模的真实失败样本校准任务 13 特征精度,同时保持所有判定为
   顾问式、LLM 回退仍需 opt-in。
2. 向 App 团队申请稳定的语义 ID,用于文本字段、日期分组和底部导航。
3. 在模拟器上校验 iOS locator 集合,并把原生 picker 的差异隔离在 Page
   层。
4. 仅在保留一条干净的首次安装冒烟路径之后,再引入更快的 state seeding。
5. 通过带鉴权的构建工件服务发布 APK,使移动端 E2E 可以在每次可信分支
   构建上运行。
6. 仅在多团队需要并发编辑、权限和审计历史时,才把本地 Excel 登记册
   替换为带评审的测试管理 API。

主要的经验是:当断言能跨屏幕追踪数据时,小套件才真正有价值。7 个
场景配合确定性状态和强力的下游检查,比一大堆浅层的 UI 脚本提供更
多信号。
