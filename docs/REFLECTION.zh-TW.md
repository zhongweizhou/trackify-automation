# 專案覆盤

## 成果

本次交付涵蓋了兩條核心路徑下共 7 個 Android BDD 場景:

| 路徑 | 場景 | 結果 |
|---------|-----------|--------|
| Add Transaction | Expense、Income、Transfer、空金額、自訂類別 | 5 個通過 |
| Transactions | 類型篩選、依歷史日期分組 | 2 個通過 |

每個業務場景都完成了首次執行 setup,成功的 Add Transaction 場景同時
校驗了 Transactions 中對應的列與 Home 上的月度彙總。最近一次本地
Allure 執行結果為 7 通過、0 失敗、0 略過。

## 做得好的地方

### 清晰的層級歸屬

把 Gherkin、Steps、Flows、Pages 與 locator YAML 分離,讓失敗被定位在
單一層。例如,一次鍵盤互動變更只影響一個 Page 方法,而不是所有場景。
交易彙總計算也因此得以作為 Flow 行為而非 UI 選擇器邏輯來測試。

### 確定性的 Setup

在每個場景前清空套件資料有性能成本,但讓 onboarding、自訂類別建立、
彙總基準都可重複。顯式輸入使用者名稱、貨幣、預算與簡訊偏好,也是測
試真正的首次執行路徑,而不是用 Skip 繞過。

### 越過儲存確認的斷言

成功的場景不會在 Add Transaction 表單關閉時就停下。它們會在
Transactions 中校驗日期、金額、類別、時間,然後再到 Home 上檢查收入、
支出、結餘以及預算百分比。這能抓到只靠 toast 或 Recent Transactions
斷言發現不了的整合問題。

### 失敗證據

Allure 使用原始的 BDD Feature 與 Scenario 名稱。call 階段失敗時,本
地 PNG 會被儲存並附加到結果裡,不包裹測試本體,也不隱藏原始 traceback。

## 比預期更困難的地方

### Flutter 的可存取性語意

部分控制項暴露了較強的 accessibility ID,另一些則被合併進大型語意節
點,或者只暴露位置性的文字欄位。套件因此包含受限的 XPath 兜底。它們
被隔離在 YAML 中,但對應用佈局變更比穩定的語意 ID 更敏感。

### 鍵盤狀態

Android 鍵盤會同時擋住 onboarding 的預算滑桿和 Add Transaction 的控制
項。在金額、備註、標籤與類別名輸入後送出 IME Done 動作,比點擊任意座
標更可靠。

### 橫向類別

`New` 類別入口位於橫向清單的最右端。建立 `baby cost` 需要受控的滑
動、類別建立、Back 導覽,以及選取一個可能因為固定寬度只顯示 `baby`
的 chip。

### 轉場競態

當一個場景內連續新增兩筆交易時,第二次的 Home 捷徑入口可能命中關閉
中表單上同名的 tab。在儲存成功後等待 Home 可見,消除了這個競態,讓
連續交易變得可靠。

## 目前的限制

- 每個裝置 worker 都是序列的;矩陣模式增加的是裝置級並行,並不能在
  同一目標上同時跑多個 Appium 連線。
- 全套件重置讓每個場景比針對性的 state fixture 慢。
- iOS locator 與全部 7 個場景已在執行 iOS 26.5 的 iPhone 17 模擬器上
  校驗;其他裝置 profile 與語系設定尚未驗證。
- 照片附件與更廣的 Settings/Analytics 行為不在 7 個場景範圍內。
- GitHub 上的 E2E 需要透過 `TRACKIFY_APK_URL` secret 提供可外部下載的
  APK;沒有它時,CI 只跑單元測試與 collection。
- 任務 14 是刻意單向的:Excel 更新託管的 Scenario 區塊,但不把結果回
  寫,locator 也不會自癒。
- 任務 13 是顧問式而非自主的。MiniMax 相容回退已經過實測,但嚴格的
  JSON 失敗仍會降級為 `Unknown`;在調整閾值或策略前需要更多真實失敗
  樣本。
- `summary.xlsx` 的產生在任務 11 中被刻意延後。

## AI 協助

AI 幫助審閱倉庫、把規格翻譯為 BDD 與 Page Object、推理 Flutter/Appium
的 locator 行為,並縮短除錯週期。每一次建議的互動都會用真實模擬器
對照檢查。最有用的回饋來自真實的失敗:類別導覽、鍵盤狀態、日期/時
間持久化、儲存後的競態 —— 這些都需要從執行中的應用取得證據,而不是
照單全收產生的程式碼。

任務 13 內部也遵循同樣的邊界:已知失敗型態優先使用確定性特徵,只有
證據模糊時才使用 MiniMax。實測已驗證了鑑權、TLS 信任、請求/回應相
容性以及安全降級,但模型建議仍是附加在原始失敗上的假設,不是自動判
定。

## 後續改進

1. 用更大規模的真實失敗樣本校正任務 13 特徵精度,同時保持所有判定為
   顧問式、LLM 回退仍需 opt-in。
2. 向 App 團隊申請穩定的語意 ID,用於文字欄位、日期分組與底部導覽。
3. 在模擬器上校驗 iOS locator 集合,並把原生 picker 的差異隔離在 Page
   層。
4. 僅在保留一條乾淨的首次安裝冒煙路徑後,再引入更快的 state seeding。
5. 透過帶鑑權的構建工件服務發佈 APK,使行動端 E2E 可以在每次可信分
   支建置上執行。
6. 僅在多團隊需要並發編輯、權限與稽核歷史時,才把本地 Excel 登記冊
   替換為帶審閱的測試管理 API。

主要的經驗是:當斷言能跨螢幕追蹤資料時,小套件才真正有價值。7 個
場景配合確定性狀態與強力的下游檢查,比一大堆淺層的 UI 腳本提供更多
訊號。
